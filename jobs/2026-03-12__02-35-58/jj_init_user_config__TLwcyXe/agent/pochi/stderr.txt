# Initialize a Jujutsu Repository and Configure User Identity You are a developer who has recently switched from Git to Jujutsu (jj). You have an existing project directory at `/home/user/myproject` that you want to bring under jj version control. Your goal is to initialise a jj repository in that directory and set your user identity in the global jj configuration so that commits are attributed to you. ## Background Jujutsu (jj) is a modern version-control system with first-class support for Git backends. Unlike Git, jj separates the concept of a "change" (an intention to modify the codebase) from a "commit" (a snapshot of the repo at a point in time). Every working-copy state is automatically snapshotted, so there is no explicit staging area. When you first use jj you must configure two pieces of identity information that will be embedded in every commit you author: - **user.name** – your display name - **user.email** – your e-mail address These values live in the user-level (global) configuration file, not the repository-level file. On Linux the global config is stored at `~/.config/jj/config.toml` (or the path returned by `jj config path --user`). ## Starting Environment - You are logged in as `user` with home directory `/home/user`. - The directory `/home/user/myproject` already exists and contains a single file: `README.md` with the content `# My Project`. - `jj` (version 0.38.0) is installed and available on `PATH` as `/usr/local/bin/jj`. - No jj repository has been initialised yet anywhere under `/home/user`. - No jj user-level configuration exists yet (i.e. `~/.config/jj/config.toml` does not exist). ## Your Tasks ### Task 1 — Initialise a jj repository Initialise a jj repository **with a Git backend** inside `/home/user/myproject`. The correct command to initialise a new Git-backed jj repo in an existing directory is: ``` jj git init /home/user/myproject ``` or, if you are already inside the directory: ``` cd /home/user/myproject jj git init . ``` After running this command you should see a `.jj` subdirectory created inside `/home/user/myproject`. **Verify:** Run `jj status` from inside `/home/user/myproject`. You should see output indicating that jj recognises the repository, for example: ``` The working copy is clean Working copy : <change-id> <commit-id> (empty) (no description set) Parent commit: zzzzzzzz 000000000000 (empty) (no description set) ``` ### Task 2 — Set your user identity Set the following values in the **user-level** (global) jj configuration: | Key | Value | |---|---| | `user.name` | `Alex Developer` | | `user.email` | `alex@example.com` | Use `jj config set --user` for each key: ``` jj config set --user user.name "Alex Developer" jj config set --user user.email "alex@example.com" ``` You can verify the values were saved correctly with: ``` jj config get user.name jj config get user.email ``` Or list all user-level config settings: ``` jj config list --user ``` ### Verification Conditions The automated verifier will check **all** of the following: 1. The `--user` flag means the setting is stored in your home directory's config file rather than in the repository. - Unlike Git, jj does **not** require you to stage files before committing. The working copy is always automatically tracked. - There is no concept of an "index" or "staging area" in jj. - The command `jj status` is analogous to `git status`. ## Summary of Commands ```bash # 1.
